# LaLume Website

LaLume is a Canadian clean skincare brand — a **static HTML/CSS/JS site** with a partial Shopify theme structure. No framework, no build step, no package manager.

**Stack:** Vanilla HTML5 · CSS3 (single file, CSS variables) · Vanilla JS · Shopify theme (partial, not yet live)

## Development

No build tooling. Preview with any static file server:

```bash
python -m http.server 8080
# or
npx serve .
```

There are no automated tests. Validate changes by opening the relevant HTML file in a browser.

## Architecture

### Pages
- `index.html` — Hero, marquee, product showcase, FAQ
- `collections.html` — Shop/catalog
- `our-story.html` — Brand narrative
- `products/` — One HTML file per product (`niacinamide-serum`, `copper-peptide-serum`, `gold-eye-patches`, `jade-ritual-set`, `led-therapy-mask`)

### CSS (`css/style.css`)
Single file, ~916 lines. Sections ordered: keyframes → reset → typography → components → layout. Everything is driven by CSS custom properties.

Key design tokens:
- **Fonts**: Playfair Display (headings), Cormorant Garamond (elegant body), Inter (UI)
- **Colors**: forest green `#2D4A40`, gold `#B8924A`, blush `#EAD6C4`
- **Spacing**: `clamp()` throughout for fluid responsive values
- **Animations**: `blobMorph`, `floatBadge`, `clipRevealUp`, `marqueeLeft`, `shimmerGold`

### JavaScript (`js/`)
- `app.js` — Custom cursor, cart management, nav scroll behavior, magnetic buttons (IIFE pattern)
- `hero-canvas.js` — Canvas-based hero visualization
- `three-hero.js` — Three.js integration (progressive enhancement, optional)

No bundler. Script tags load files directly.

### Shopify Theme (`shopify-theme/`)
Scaffold only — `layout/`, `templates/`, `snippets/` are empty. `config/settings_schema.json` holds theme metadata; `locales/en.default.json` holds i18n strings. Integration is planned, not live.

# Ruflo — Agent Coordination

## Rules

- Do what has been asked; nothing more, nothing less
- NEVER create files unless absolutely necessary — prefer editing existing files
- NEVER create documentation files unless explicitly requested
- NEVER save working files or tests to root — use `/src`, `/tests`, `/docs`, `/config`, `/scripts`
- ALWAYS read a file before editing it
- NEVER commit secrets, credentials, or .env files
- Keep files under 500 lines
- Validate input at system boundaries

## Agent Comms (SendMessage-First Coordination)

Named agents coordinate via `SendMessage`, not polling or shared state.

```
Lead (you) ←→ architect ←→ developer ←→ tester ←→ reviewer
              (named agents message each other directly)
```

### Spawning a Coordinated Team

```javascript
// ALL agents in ONE message, each knows WHO to message next
Agent({ prompt: "Research the codebase. SendMessage findings to 'architect'.",
  subagent_type: "researcher", name: "researcher", run_in_background: true })
Agent({ prompt: "Wait for 'researcher'. Design solution. SendMessage to 'coder'.",
  subagent_type: "system-architect", name: "architect", run_in_background: true })
Agent({ prompt: "Wait for 'architect'. Implement it. SendMessage to 'tester'.",
  subagent_type: "coder", name: "coder", run_in_background: true })
Agent({ prompt: "Wait for 'coder'. Write tests. SendMessage results to 'reviewer'.",
  subagent_type: "tester", name: "tester", run_in_background: true })
Agent({ prompt: "Wait for 'tester'. Review code quality and security.",
  subagent_type: "reviewer", name: "reviewer", run_in_background: true })

// Kick off the pipeline
SendMessage({ to: "researcher", summary: "Start", message: "[task context]" })
```

### Patterns

| Pattern | Flow | Use When |
|---------|------|----------|
| **Pipeline** | A → B → C → D | Sequential dependencies (feature dev) |
| **Fan-out** | Lead → A, B, C → Lead | Independent parallel work (research) |
| **Supervisor** | Lead ↔ workers | Ongoing coordination (complex refactor) |

### Rules

- ALWAYS name agents — `name: "role"` makes them addressable
- ALWAYS include comms instructions in prompts — who to message, what to send
- Spawn ALL agents in ONE message with `run_in_background: true`
- After spawning: STOP, tell user what's running, wait for results
- NEVER poll status — agents message back or complete automatically


### Agent Routing

| Task | Agents | Topology |
|------|--------|----------|
| Bug Fix | researcher, coder, tester | hierarchical |
| Feature | architect, coder, tester, reviewer | hierarchical |
| Refactor | architect, coder, reviewer | hierarchical |
| Performance | perf-engineer, coder | hierarchical |
| Security | security-architect, auditor | hierarchical |

### When to Swarm
- **YES**: 3+ files, new features, cross-module refactoring, API changes, security, performance
- **NO**: single file edits, 1-2 line fixes, docs updates, config changes, questions

### 3-Tier Model Routing

| Tier | Handler | Use Cases |
|------|---------|-----------|
| 1 | Agent Booster (WASM) | Simple transforms — skip LLM, use Edit directly |
| 2 | Haiku | Simple tasks, low complexity |
| 3 | Sonnet/Opus | Architecture, security, complex reasoning |
